Okay. One morning we woke up on an uninhabited island.

We have nothing but a laptop with 64-bit Windows 10 and Excel 2016.
Don't worry, that'll be enough to build any 3D model in any detail you
may need. Here I‘ll show you how.

The main idea is to use common WinAPI libraries. From this point, it
doesn’t matter which language to use for calling API functions. Excel
2016 offers VBA. Not bad!

To create a 3D environment we’ll use OpenGL as well.

This is all we need to know before the start.

# Part 1

First off, we have to write the Hello World framework.

Launch the Excel App, create a new Excel book, and press Alt+F11 to open
the VBA. In the VBA window menu, we choose "Insert" -> "Module". Now we
have a white field with a lonely text: "Option Explicit". This will be
our workspace.

To show the "Hello World" greeting, we can use the built-in VBA
function: MsgBox ("Hello World") but our primary goal is to explore
how WinAPI works in the VBA environment so we use the "User32" library
instead.

Yes, the "User32" and "Kernel32", as well as many other libraries,
are accessible from VBA!

Maybe that's the reason why some people consider VBA unsafe.

Here's how to declare the API functions in VBA:

> Declare PtrSafe Function MessageBoxW Lib "User32" (ByVal hWnd As
> LongPtr, ByVal lpText As LongPtr, ByVal lpCaption As LongPtr, ByVal
> uType As LongPtr) As LongPtr

We must use the PtrSafe keyword in combination with the LongPtr data
type to provide compatibility with 32-bit and 64-bit platforms.

All arguments must be declared ByVal (by value) and must have the same
size: 4 bytes for 32 bits and 8 bytes for 64 bits, respectively.

The following table helps understanding the principles of data size
correspondence.

+-------------+-------------+-------------+-------------+-------------+
| WinAPI data | Example     | Size in     | VBA         | Notes       |
| type        |             | memory      | variable    |             |
|             |             |             | type        |             |
+:===========:+:===========:+:===========:+:===========:+:===========:+
| Arguments   | ByVal msg   | 4 bytes for | Long for 32 |             |
| passed to   | As LongPtr  | 32 bits     | bits        |             |
| functions   |             |             |             |             |
|             |             | 8 bytes for | LongLong    |             |
|             |             | 64 bits     | for 64 bits |             |
|             |             |             |             |             |
|             |             |             | LongPtr for |             |
|             |             |             | both        |             |
+-------------+-------------+-------------+-------------+-------------+
| Handles,    | hWnd,       | 4 bytes for | Long for 32 |             |
| pointers    | hMenu,      | 32 bits     | bits        |             |
|             |             |             |             |             |
|             | lpText,     | 8 bytes for | LongLong    |             |
|             | lpWndClass  | 64 bits     | for 64 bits |             |
|             |             |             |             |             |
|             |             |             | LongPtr for |             |
|             |             |             | both        |             |
+-------------+-------------+-------------+-------------+-------------+

We also must use the Unicode versions of the functions. MessageBoxW
stands for Unicode and MessageBoxA is for ANSI. The difference is that
Unicode takes 2 bytes per symbol while ANSI takes only 1 byte. However,
VBA works correctly only with the 2-byte symbol representation.

As we can see, three of arguments are addresses in memory: hWnd, lpText,
and lpCaption. The fourth argument is a constant.

There is no Internet on our uninhabited island so we can't download
header files such as "windows.h".

So we need to declare certain constants like this:

> Public Const MB_OK = 0
>
> Public Const MB_ICONINFORMATION = &H40

Our message box will contain only one button and it will show the info
mark.

After all declarations, we can call the function.

According to VBA syntax, executable code must be contained in a
procedure:

> Sub HelloWorldMessageProc()
>
>     Call MessageBoxW(0, StrPtr("Hello World"), StrPtr("VBA WinAPI
> Demo"), MB_OK Or MB_ICONINFORMATION)
>
> End Sub

Since there is no main window yet, hWnd = 0.

The StrPtr function returns the addresses of text strings in memory.
VBA's environment has the special containment for text strings. So don't
confuse it with the VarPtr function that is used for getting addresses
of other types of variables.

If everything has been done right then by pressing F5 we can get a
little message box named "VBA WinAPI Demo" with a message "Hello World":

height="1.6252263779527558in"}

Congrats! We've just proven that VBA can call WinAPI functions.

Now we can use the full range of this powerful toolset.

The WinMain function in VBA looks familiar but there is one little
trick.

To register the main window class, we need to load the handle (i.e.
address) of WndProc to the wcx structure. But for some reason, the VBA's
inner AddressOf operator cannot be used in assignment operations
immediately.

<https://citforum.ru/programming/vb/vba_winapi/7.shtml>

The decision is to pass the WndProc's address to an extra procedure that
will return it back:

> Function WinMain(...)
>
>     ...
>
>     wcx.lpfnWndProc = GetAddr(AddressOf WndProc)
>
>     ...
>
> End Function
>
> Function GetAddr(ByVal lpProc As LongPtr) As LongPtr
>
>     GetAddr = lpProc
>
> End Function

dummies that we'll fill in the Part 3:

# Part 4

We have a window, we have OpenGL, and we have a colourful triangle. But
the title of this article promised "any 3D model in any detail". A
triangle is not enough. Let's load some real data.

Our "uninhabited island" is actually the site of the Chernobyl Nuclear
Power Plant. The data we have is a 48x48 matrix of cell values – the
RBMK-1000 reactor layout (each cell represents a fuel channel, control
rod, or graphite block). In Excel, these values are stored on a sheet.
We read them into a two-dimensional byte array:

> Public RBMK(0 To 47, 0 To 47) As Byte
>
> Sub Start()
>
> Dim x As Byte, y As Byte
>
> Dim u As Byte, v As Byte
>
> u = 4: v = 51
>
> For y = 0 To 47
>
> For x = 0 To 47
>
> RBMK(x, y) = ActiveSheet.Cells(v, u).Value
>
> u = u + 1
>
> Next x
>
> u = 4: v = v - 1
>
> Next y
>
> Call WinMain(0, 0, 0, 0)
>
> End Sub

Now we have the reactor map. Each cell will be drawn as a small 3D cap
(a kind of truncated pyramid). The colour of the cap depends on the
value: 1 – gray, 2 – green, 3 – yellow, 4 – red, 5 – blue. This is our
"model".

In `DrawObject`, we simply iterate over the array, translate to the
correct position, set the material (ambient + diffuse), and call a
small `DrawCap` routine that draws 6 quads (top and four sides) using
immediate mode `glBegin(GL_QUADS)`. For 48x48 = 2304 caps, that's about
2304 * 24 = 55,296 vertices per frame. On modern hardware, that's fine
– VBA can handle it because most of the work is done by the GPU.

But we are not satisfied with just a static model. We want to fly around
it – tilt, turn, zoom, walk through it. This requires a camera that can
be rotated arbitrarily without gimbal lock. And we want to rotate the
model itself independently. That's where **quaternions** come in.

# Part 5

In Part 4 we drew the reactor layout. But the view was static. To make
it interactive, we need to process mouse and keyboard input. The classic
approach uses Euler angles (pitch, yaw, roll). But Euler angles suffer
from gimbal lock – you lose one degree of freedom when pitch reaches
±90°. That's unacceptable for a serious viewer. Professional game
engines use **quaternions**.

A quaternion is a four-dimensional number (x, y, z, w) that represents
any 3D rotation without singularities. Multiplying two quaternions gives
a combined rotation. Rotating a vector by a quaternion is a few
multiplications – very fast.

We define a quaternion type in VBA:

> Type QUATERNION
>
> x As Single
>
> y As Single
>
> z As Single
>
> w As Single
>
> End Type

Then we write functions to:

- Build a quaternion from an axis and angle (`QuatFromAxisAngle`).

- Multiply two quaternions (Hamilton product) – `MulQbyQ`.

- Convert a quaternion to a 4x4 rotation matrix for OpenGL (`CvtQtoM`).

- Rotate a 3D vector by a quaternion (`RotateVectorByQuaternion`).

The logic of our viewer is simple:

- **Left mouse button** drag – rotates the **model** around its local
  axes (right-multiply quaternion).

- **Right mouse button** drag – rotates the **camera** around world
  axes (left-multiply quaternion).

- **Middle button** drag – moves the camera up/down and left/right (in
  camera space).

- **Mouse wheel** – moves the camera forward/backward.

- **Arrow keys** – strafe and move forward/backward in camera space.

- **W, S, A, D, Q, E** – rotate the model (tilt, turn, roll) using
  quaternions.

- **Shift** – speed boost.

- **Space / middle double-click** – reset view.

All positions and orientations are stored in global variables:

> Public camPos As POINT3D
>
> Public qModelRot As QUATERNION
>
> Public qCamRot As QUATERNION

Every time the user interacts, we update these variables and
call `UpdateModelview`, which rebuilds the OpenGL matrix from scratch:

> Function UpdateModelview()
>
> Call glMatrixMode(GL_MODELVIEW)
>
> Call glLoadIdentity
>
> ' 1) camera orientation (world rotation)
>
> Call CvtQtoM(qCamRot)
>
> Call glMultMatrixf(lp_mtxRotationBuffer)
>
> ' 2) camera position and a fixed tilt (300 degrees about X)
>
> Call glTranslatef(camPos.x, camPos.y, camPos.z)
>
> Call glRotatef(300, 1, 0, 0)
>
> ' 3) model orientation
>
> Call CvtQtoM(qModelRot)
>
> Call glMultMatrixf(lp_mtxRotationBuffer)
>
> ' 4) a fixed 20-degree turn of the whole model
>
> Call glRotatef(20, 0, 0, 1)
>
> isRefreshed = 0
>
> End Function

No gimbal lock, no matrix drift – just smooth, predictable rotation.
(For efficiency, we also split matrix updates: the projection matrix is
rebuilt only on window resize, while the modelview matrix is updated on
every interaction. The full code in the attachment includes both
`UpdateProjection` and `UpdateModelview`.)

# Part 6

The result is a fully interactive 3D viewer of the RBMK reactor layout,
running entirely inside Excel 2016 on a 64-bit Windows 10 laptop. You
can orbit around the model, tilt the camera, zoom in and out, and rotate
the reactor itself with the keyboard - all at a solid frame rate (60+
FPS on typical hardware).

The full code is about 1500 lines of VBA, including all WinAPI
declarations, OpenGL bindings, the message loop, quaternion math, and
the drawing routines. It uses nothing but the standard Windows libraries
- no extra DLLs, no third-party tools, no .NET. Just Excel and the
operating system.

    rendering.

    pointer arithmetic.

    visualisations.
